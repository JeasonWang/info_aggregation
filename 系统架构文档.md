# 信息聚合系统（Info_aggregation）系统架构文档

> 版本：1.0.0  
> 日期：2026-04-17  
> 项目名称：Info_aggregation — 信息聚合系统

---

## 一、系统概述

信息聚合系统是一个基于 Python 的多渠道信息采集、清洗、存储与查询平台。系统从微博、今日头条、小红书、东方财富网、路透社、CSDN、掘金、博客园、36氪、知乎等 10 个渠道精准爬取信息，按 5 大分类（热点事件、经济数据、国际大事、科技动向、AI大模型动向）进行规范存储，并通过 RESTful API 对外提供数据查询服务。

### 1.1 核心目标

- **精准爬取**：覆盖 10 个主流信息渠道，按分类差异化爬取频率
- **规范存储**：三表结构设计，唯一约束防重复，索引优化查询效率
- **前端展示**：FastAPI 提供 7 个 RESTful API 端点，支持分页、筛选、搜索
- **稳定运行**：定时任务自动调度，异常重试机制，全链路日志监控

### 1.2 技术栈

| 层次 | 技术选型 | 版本要求 | 说明 |
|------|---------|---------|------|
| Web 框架 | FastAPI | ≥0.104.0 | 高性能异步 API 框架 |
| ASGI 服务器 | Uvicorn | ≥0.24.0 | 生产级 ASGI 服务器 |
| ORM | SQLAlchemy | ≥2.0.0 | Python 标准 ORM 框架 |
| HTTP 客户端 | Requests | ≥2.31.0 | 爬虫 HTTP 请求库 |
| 定时任务 | APScheduler | ≥3.10.0 | 后台定时任务调度器 |
| 数据库 | SQLite / MySQL | - | 默认 SQLite（开箱即用），可切换 MySQL |

### 2.2 模块组成与职责

| 模块 | 目录 | 职责 | 核心类/函数 |
|------|------|------|------------|
| **爬虫模块** | `crawlers/` | 分渠道实现爬取逻辑，抽象基类统一接口 | `BaseCrawler`, `WeiboCrawler` 等 10 个子类 |
| **爬虫注册中心** | `crawlers/registry.py` | 管理爬虫实例的注册与获取 | `CrawlerRegistry` |
| **数据清洗模块** | `cleaners/` | 数据去重、HTML标签清除、字段截断、格式规范化 | `clean_info_list()`, `clean_info_item()` |
| **数据库存储模块** | `database/` | ORM 模型定义、会话管理、表结构初始化 | `Category`, `Channel`, `Info`, `get_session()` |
| **定时任务模块** | `scheduler/` | 按分类配置定时爬取任务，失败重试 | `setup_scheduler()`, `crawl_by_category()` |
| **API 模块** | `api/` | RESTful 接口，信息查询/统计/手动触发爬取 | FastAPI `app`, 7 个端点 |
| **日志与监控模块** | `main.py` + 各模块 | 双通道日志（控制台+文件），全链路异常捕获 | `setup_logging()` |
| **全局配置** | `config.py` | 集中管理所有配置项，支持环境变量覆盖 | 常量定义 |
| **数据初始化** | `sql/init_data.py` | 建表、初始化分类/渠道、插入模拟数据 | `init_all_data()` |

### 2.3 模块交互逻辑

```
启动流程:
  main.py
    ├── [1/4] init_all_data()     → database/ 创建表 + sql/ 插入初始数据
    ├── [2/4] register_all_crawlers() → crawlers/registry.py 注册 10 个爬虫
    ├── [3/4] setup_scheduler()   → scheduler/ 配置 5 个定时任务
    └── [4/4] uvicorn.run(app)    → api/ 启动 FastAPI 服务

定时爬取流程:
  Scheduler 触发
    → crawl_by_category(category_name)
      → 查询数据库获取该分类下的活跃渠道
      → 遍历 crawler_registry 中匹配的爬虫
        → crawler.safe_crawl()       [爬虫模块: 请求→解析→返回原始数据]
        → clean_info_list(raw_items) [清洗模块: 去重+规范化]
        → _save_crawled_data()       [数据库模块: 去重写入]

API 查询流程:
  客户端请求
    → FastAPI 路由分发
      → get_session() 获取数据库会话
      → SQLAlchemy 查询
      → to_dict() 序列化
      → 返回 JSON 响应
```

**索引设计：**

| 索引名 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| uq_source_channel | (source_id, channel_id) | UNIQUE | 防止同一渠道重复数据 |
| idx_category_id | category_id | NORMAL | 加速按分类查询 |
| idx_channel_id | channel_id | NORMAL | 加速按渠道查询 |

| idx_created_at | created_at | NORMAL | 加速按创建时间排序 |

### 3.5 设计考量

- **数据冗余与查询效率平衡**：info 表同时存储 category_id 和 channel_id，避免多表 JOIN 查询分类和渠道信息；通过 `to_dict()` 方法在序列化时自动填充 category_name 和 channel_name
- **防止单表过大**：info 表使用索引优化查询，逻辑删除字段 is_deleted 支持软删除，后续可做数据归档
- **去重机制**：双层去重——数据库层 UNIQUE 约束 + 应用层 `clean_info_list()` 按 source_id 去重 + `_save_crawled_data()` 查询已有记录

## 四、爬虫模块设计

### 4.1 爬虫抽象基类（BaseCrawler）

所有渠道爬虫必须继承 `BaseCrawler` 并实现 `crawl()` 方法。基类提供：

| 方法 | 功能 |
|------|------|
| `_get_random_ua()` | 从 5 个 User-Agent 中随机选取，防止反爬 |
| `_build_headers()` | 构建标准请求头（含随机 UA、Accept、Language 等） |
| `fetch(url, params, headers)` | HTTP GET 请求，内置 3 次重试，间隔 5 分钟 |
| `fetch_json(url, params, headers)` | HTTP GET 请求并返回 JSON 解析结果 |
| `crawl()` | **抽象方法**，子类必须实现，返回标准化信息列表 |
| `safe_crawl()` | 安全包装 crawl()，捕获异常返回空列表 |

### 4.2 爬虫返回数据格式

每个爬虫的 `crawl()` 方法必须返回如下格式的字典列表：

```python
{
    "source_id": str,       # 来源唯一标识（用于去重）
    "title": str,           # 标题（≤40字）
    "content": str,         # 内容/事件简介（≤300字）
    "source_url": str,      # 来源URL
    "event_time": datetime, # 事件时间
    "core_entity": str,     # 核心主体/人物
    "location": str,        # 地点（可选，热点/国际类使用）
    "indicator_name": str,  # 指标名称（经济数据类使用）
    "indicator_value": str, # 指标数值（经济数据类使用）
}
```

|------|------|------|---------|---------|---------|

| 东方财富网 | `EastmoneyCrawler` | 经济数据 | 1小时 | 东方财富行情API | 指标名称、数值、更新时间、来源 |

### 4.4 反爬策略

| 策略 | 实现方式 | 配置项 |
|------|---------|--------|
| User-Agent 随机化 | 5 个主流 UA 随机轮换 | `CRAWLER_USER_AGENTS` |
| 请求间隔控制 | 每次请求后随机等待 0.5~2.0 秒 | `fetch()` 内置 |
| 失败重试 | 最多重试 3 次，间隔 5 分钟 | `CRAWLER_RETRY_TIMES`, `CRAWLER_RETRY_INTERVAL` |
| 请求超时 | 单次请求超时 15 秒 | `CRAWLER_REQUEST_TIMEOUT` |
| Referer 伪装 | 各爬虫设置对应来源 Referer | 各爬虫 `crawl()` 内设置 |

### 4.5 爬虫注册中心（CrawlerRegistry）

采用注册模式管理爬虫实例，解耦爬虫实现与调度逻辑：

```python
# 注册爬虫
crawler_registry.register("weibo", WeiboCrawler())

# 获取爬虫
crawler = crawler_registry.get("weibo")

# 获取所有已注册爬虫
all_crawlers = crawler_registry.get_all()
```

## 五、数据清洗模块设计

### 5.1 清洗流程

```
原始数据列表
    │
    ▼
clean_info_list(items)
    │
    ├── 遍历每条数据
    │     │
    │     ▼
    │   clean_info_item(item)
    │     │
    │     ├── clean_title()       → 去HTML标签、去空白、截断40字
    │     ├── clean_content()     → 去HTML标签、去空白、截断300字
    │     ├── clean_source_url()  → 验证URL格式合法性
    │     ├── source_id 校验      → 无source_id则丢弃
    │     ├── event_time 规范化   → 字符串转datetime，缺省取当前时间
    │     └── 字段截断            → core_entity/location/indicator_* 截断100字
    │
    ├── 按 source_id 去重（内存层）
    │
    └── 返回清洗后列表
```

### 5.2 清洗规则

| 字段 | 清洗规则 | 不合格处理 |
|------|---------|-----------|
| title | 去HTML标签、合并空白、截断40字 | 为空则丢弃整条 |
| content | 去HTML标签、合并空白、截断300字 | 为空则保留（允许无内容） |
| source_url | 验证 http/https 开头 | 不合法则置空 |
| source_id | 去首尾空白 | 为空则丢弃整条 |
| event_time | 字符串解析为 datetime | 解析失败取当前时间 |
| core_entity | 截断100字 | - |
| location | 截断100字 | - |
| indicator_name | 截断100字 | - |
| indicator_value | 截断100字 | - |

## 六、定时任务设计

### 6.1 任务配置

| 任务ID | 任务名称 | 执行频率 | 触发条件 | 目标分类 | 涉及渠道 |
|--------|---------|---------|---------|---------|---------|
| crawl_hot | 热点事件爬取 | 每30分钟 | 固定间隔 | 热点事件 | 微博、今日头条、小红书 |
| crawl_economy | 经济数据爬取 | 每60分钟 | 固定间隔 | 经济数据 | 东方财富网 |
| crawl_international | 国际大事爬取 | 每120分钟 | 固定间隔 | 国际大事 | 路透社 |
| crawl_tech | 科技动向爬取 | 每120分钟 | 固定间隔 | 科技动向 | CSDN、掘金、博客园 |
| crawl_ai | AI大模型动向爬取 | 每120分钟 | 固定间隔 | AI大模型动向 | 36氪、知乎 |

### 6.2 调度器配置

- **调度器类型**：`BackgroundScheduler`（后台运行，不阻塞主线程）
- **触发器类型**：`IntervalTrigger`（固定间隔触发）
- **最大并发实例**：`max_instances=1`（同一任务不并发执行）
- **错过执行宽限时间**：`misfire_grace_time=300`（5分钟内仍可执行）

### 6.3 失败重试机制

```
定时任务触发
    │
    ▼
crawl_by_category(category_name)
    │
    ├── 查询该分类下的活跃渠道
    │
    └── 遍历每个渠道爬虫
          │
          ├── crawler.safe_crawl()     ← 内部捕获异常，返回空列表
          │     │
          │     └── crawler.crawl()
          │           │
          │           └── self.fetch()  ← 内置3次重试
          │                 │
          │                 ├── 第1次失败 → 等待5分钟 → 重试
          │                 ├── 第2次失败 → 等待5分钟 → 重试
          │                 └── 第3次失败 → 记录错误日志，抛出异常
          │
          ├── clean_info_list()         ← 清洗去重
          │
          └── _save_crawled_data()      ← 写入数据库（应用层去重）
                │
                └── 异常 → session.rollback() + 记录错误日志
```

### 6.4 手动触发

除定时自动执行外，系统提供 API 端点 `POST /api/crawl/trigger?channel_code=xxx` 支持手动触发指定渠道的爬取任务。

## 十、扩展性设计

### 10.1 新增爬取渠道

只需 3 步即可新增一个渠道：

**步骤 1**：在 `crawlers/` 目录下创建新爬虫文件，继承 `BaseCrawler`

```python
# crawlers/new_channel.py
from .base import BaseCrawler

class NewChannelCrawler(BaseCrawler):
    def __init__(self):
        super().__init__("new_channel", "新渠道名称")

    def crawl(self) -> list:
        # 实现爬取逻辑，返回标准化信息列表
        results = []
        # ... 爬取代码 ...
        return results
```

**步骤 2**：在 `main.py` 的 `register_all_crawlers()` 中注册

```python
from crawlers.new_channel import NewChannelCrawler
crawler_registry.register("new_channel", NewChannelCrawler())
```

**步骤 3**：在 `sql/init_data.py` 的 `init_channels()` 中添加渠道记录

```python
{"name": "新渠道名称", "code": "new_channel", "base_url": "https://...", 
 "category": CATEGORY_XXX, "interval": 60}
```

无需修改调度器、API、清洗模块等任何其他代码。

### 10.2 扩展信息分类

通过分类表动态配置，无需修改代码结构：

**步骤 1**：在 `config.py` 中添加新分类常量

```python
CATEGORY_NEW = "新分类名称"
```

**步骤 2**：在 `sql/init_data.py` 的 `init_categories()` 中添加分类记录

**步骤 3**：将对应渠道的 `category` 指向新分类

分类表通过数据库记录管理，支持运行时动态增删，无需重启系统。

### 10.3 切换数据库

系统默认使用 SQLite（零配置开箱即用），切换到 MySQL 只需设置环境变量：

```bash
export DB_TYPE=mysql
export DB_HOST=127.0.0.1
export DB_PORT=3306
export DB_USER=root
export DB_PASSWORD=your_password
export DB_NAME=info_aggregation
```

或在 `config.py` 中直接修改默认值。

### 10.4 扩展 API

在 `api/__init__.py` 中添加新的路由端点即可，FastAPI 自动生成 OpenAPI 文档。
